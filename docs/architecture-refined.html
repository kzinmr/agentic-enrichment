機序図解機序

通話文字起こしに対して「仮説 intent & query の幅だし」と「導出スキーマの誘導」を交互に提案・検証する enrichment ループの内部機序とデータフローを図解する。実行結果の可視化は 実行計器盤 (sample_dashboard.html) を、 運用手順は README を参照。

目次 壱全体データフロー 弐5つのLLM呼び出し 参結合の仕組み 四evidenceゲート 伍決定論実行エンジン 六2系統のフィードバック 七成果物ファイルマップ 八指標の読み方 九可視化機構
第壱節

全体データフロー

1 イテレーションは 5 段で構成される。意味判断は LLM（朱の段）、 実行と検証は決定論（藍の段）に分離されており、 各段の出力はすべて JSON artifact として残る。

図1: 1イテレーションの機序。朱=LLM意味判断 / 藍=決定論 / 黄土=運用履歴の合流
第弐節

5つのLLM呼び出し

semantic な判断は prompts.py に定義された 5 種類のみ。 すべて JSON mode で呼ばれ、コード側 validator が ValueError を投げると エラー文を添えて最大 3 回の修復リトライが走る（ヒューリスティック fallback は持たない）。 プロンプトはコード定数であり、変更は git の PR ＝人間レビューを必ず通る。

prompt_id役割主な入力validator が強制する契約
ideation.intent_queries@v1仮説 intent & query の幅だし 業務プロファイル / コーパス概況 / 前回 intents / signals intent_id の snake_case と安定性、operation 3種、クエリのプレースホルダ禁止
induction.schema@v1導出スキーマの誘導 intents+queries / 文字起こしサンプル / 前回スキーマ / signals serves_intents 必須、予約語排除、enum への not_mentioned 注入
extraction.call_fields@v1通話1件のフィールド抽出 field_specs / その通話の chunks evidence_chunk_ids ⊂ 提示 chunk、語彙外値の矯正と記録
query.compile@v1クエリ→実行プラン変換 クエリ / カタログ(active+メタ列) / corpus_stats 未知フィールド・語彙外値・プレースホルダ日付の拒否
trial.judge@v1実行結果の回答可否判定 クエリ / プラン / 実行結果(行・集計・根拠) failure_modes の閉集合、field_requests / query_revision の正規化

旧プロトタイプ (cu-agent-rlm / qu-agent-rlm) の retrieval controller・reranker・typed subagent contract は撤廃した。subagent を再導入するのは「別モデル・別プロセス・別SLA / 非同期 worker / プロセス境界での権限分離 / 独立デプロイ」のいずれかが実需になったときのみ。

第参節

結合の仕組み — intent ⇄ field

本デモの中心主張は「2つの成果物（intent & query / スキーマ）が 有機的に結合し、毎イテレーション検証される」こと。結合は宣言と観測の二重で担保される。

宣言 (静的) 全フィールドは serves_intents（支える intent_id のリスト）を必須で宣言する。 既知 intent に紐づかないフィールドは validator が機械的に落とし、 schema.json の dropped_fields に記録される。
観測 (動的) 全仮説クエリを実際に compile → 実行 → judge し、プランが参照したフィールドを intent 別と全体の両方で集計する（by_intent / used_by_plans）。 「宣言上は支えているが、その intent のクエリでは一度も使われていない」状態が観測できる。
警告 (自動) ①active な支えフィールドが無い ②intent 内クエリでフィールド未使用 ③高 priority で回答率 50% 未満 — のいずれかに該当する intent は 結合不全 (failing coupling) としてレポートとダッシュボードに明示される。
還流 (次周) judge は schema 側への field_requests と query 側への query_revision の両方向の改善提案を出し、次イテレーションの ideation / induction の双方に渡る。
第四節

evidence ゲート — 根拠なき値は昇格しない

旧プロトタイプから継承した最重要の安全装置。LLM が「それっぽい値」を出しただけでは クエリ基盤に乗らない。

白名単検証 抽出値の evidence_chunk_ids は、その通話で LLM に提示した chunk_id 集合内に限定。 提示外の参照は除外され validation_errors に残る（hallucination 防御）。
coverage 計測 フィールドごとに fill_rate（値が付いた率）と evidence_coverage（値のうち根拠付きの率）を計測する。
昇格判定 evidence_coverage ≥ 0.75（設定可）を満たさないフィールドは status: hold に降格し、クエリの compile カタログから外れる。 hold のままでも silver 行には値が残るため、次イテレーションで改善されれば自動復帰する。

通話レコードが元から持つメタ列（date / direction）は誘導対象ではなく 組み込みカタログフィールドとして常時 filter / aggregate に使える。PII redaction は 本ツールより上流（データ側）で実施済みであることを入力契約とする。

第伍節

決定論実行エンジン

プランの実行に LLM は関与しない。操作は 3 種に固定され、本番接続時は この 3 操作のインターフェイスを Databricks / OpenSearch 等へ差し替える。

操作実装備考
filtersilver 行のフィルタ op: eq / neq / in / contains / not_empty / gte / lte。 date は YYYY-MM 前方一致と gte/lte 範囲に対応
aggregategroup_by 件数集計 list 型は値ごとに展開、空値は not_mentioned 群に集約
search純 Python BM25 CJK は文字 bigram + 英数単語トークン（日本語対応）。 matched_terms を返し judge の説明可能性を支える

構造化フィルタがゼロヒットの場合は元クエリで BM25 フォールバック検索を行い、 judge が「データが無い」のか「抽出・語彙が外れている (improve_extraction)」のかを 区別できる候補根拠を渡す。compile 自体が失敗した場合は常に実行可能な search プランへ退避し、その発生率は compiled_structurally_rate として計測される。

第六節

2系統のフィードバック

ループ内 — judge 由来
  • field_requests（add_field / add_allowed_values / improve_extraction）
  • 未回答クエリと failure_modes
  • query_revision（keep / revise / drop）
  • hold フィールドと未使用 promoted フィールド
ループ外 — 運用履歴由来
  • production_log.jsonl（query / asked_count / outcome / operator_note）
  • クエリは weight=asked_count 付きで trials に合流
  • 頻度が高く未回答のクエリは intent 化の優先入力に
  • operator_note はスキーマ改善要求として induction へ

両系統は build_next_signals() で同一の signals 形式に正規化され、 ideation と induction の同じ入口に入る。本番接続は「検索 UI のクエリログと成否評価を この JSONL に落とす」だけでよい。評価側も asked_count を重みとして使い、 頻度 11 回の未回答運用クエリが 1 回限りの仮説クエリと同じ重みに薄まることはない。

第七節

成果物ファイルマップ

output/<run>/
├── manifest.json            run_id / prompt hash / 入力 hash / 設定 — 再現性アンカー
├── chunks.jsonl              検索可能チャンク (evidence 参照の解決先)
├── production_signals.json   運用履歴の正規化結果 (--production-log 時)
├── iter_01/ iter_02/ …
│   ├── intents.json         成果物1: 仮説 intent & query
│   ├── schema.json          成果物2: 導出スキーマ (serves_intents / dropped_fields)
│   ├── silver_records.jsonl  通話ごとの構造化行 + evidence + confidence
│   ├── quality_report.json   fill_rate / evidence_coverage / 昇格判定
│   ├── trials.json           全クエリの plan / 実行結果 / judgement
│   └── gap_signals.json      ギャップ分析 = 次イテレーションへの signals 原本
├── loop_summary.json         イテレーション推移 + usage 集計
├── trace.jsonl               全 LLM 呼び出しの構造化トレース (run_id 付き)
├── report.md                 人間向けレポート (結合マトリクス含む)
└── dashboard.html            viz/generate.py が生成する実行計器盤
第八節

指標の読み方

指標意味読み方の注意
answered_rate件数ベース回答率 ideation が難問仮説を増やすと下がるのが正常な場合がある
answered_rate_weightedasked_count 重み付き回答率 運用上の実需が満たされているかを見る主指標
production_answered_rate_weighted運用クエリのみの重み付き回答率 運用履歴フィードバックの効き目を直接示す
compiled_structurally_rateplan が search フォールバックに落ちなかった率 スキーマがクエリを構造的に支えられている度合い
evidence_sufficient_rate根拠十分と判定された率—

これらは LLM judge の自己申告に基づく。改善傾向の観測には十分だが、 品質の最終的な裏付けには hand-labeled eval fixture（期待 call_ids / 期待回答での決定論採点）が 必要であり、これは本デモの既知の残課題である。

第九節

可視化機構 (viz/)

viz/generate.py は本体パッケージに依存しない独立したデモ機構で、 run の出力ディレクトリを読んで 外部依存ゼロ・オフライン閲覧可能な 単一 HTML ダッシュボードを生成する。 

python3 viz/generate.py --run-dir output/demo --open

掲載内容: ヘッダ計器（before/after の delta 付き）、パイプライン通過量、 イテレーション指標バー、LLM 呼び出しタイムライン（prompt 種別レーン × 実時間、 修復リトライは黄枠）、intent × field 結合マトリクス（intent-local ヒートマップ + 結合不全バナー）、クエリ試行エクスプローラ（フィルタ + plan / judge / 根拠の展開）、 スキーマカード（fill / evidence バー）、残ギャップと運用フィードバック。 実出力のサンプルは sample_dashboard.html。
