基礎概念ガイド — はじめて見る方はこちら
このダッシュボードは、通話文字起こしへの「分析意図の仮説出し」と「構造化スキーマの設計」を 交互に改善するループの 1 回の実行 (run) を可視化したものです。1 イテレーションは 5 段:
① ideation — 業務プロファイルから「何を分析したいか」(= intent) と検証用クエリ仮説を提案
② induction — その intent 群の検索・分析を支える構造化フィールド (導出スキーマ) を設計
③ extraction — 全通話からフィールド値を抽出 (根拠チャンク参照が必須)
④ trials — 全クエリを実際に実行し、答えられたかを LLM judge が判定
⑤ gap analysis — 失敗・不足を集約し、次のイテレーションの ①② へ還流
intent / クエリ仮説
顧客がこのデータに対して持ちうる分析意図 (例:「予約機会損失の把握」)。 各 intent は検証用の具体的なクエリ仮説を 2〜3 件持つ。
実運用クエリ (source: production_log)
実際の運用で利用者が検索基盤に打ったクエリの履歴。w = 質問された回数 (asked_count) で、仮説クエリ (w1) より重く評価される。実運用で答えられていないクエリは intent / スキーマ見直しの最優先入力になる。
serves_intents
フィールドが「どの intent の検索・分析を支えるか」の宣言。スキーマ設計時に必須で、 どの intent も支えないフィールドは自動破棄される。スキーマカードの serves: 表示と、結合マトリクスの朱枠はこの宣言を示す。
evidence ゲート
抽出値には根拠チャンク参照が必須。根拠付き率 (evid) が 0.75 未満のフィールドは hold に降格し、クエリから使えなくなる。「LLM がそれっぽい値を出しただけ」では 分析基盤に昇格しない仕組み。
回答率 (件数) と (重み付き)
件数: 全試行のうち judge が「答えられた」と判定した割合。難しい仮説クエリが 増えると下がることがある。重み付き: 各クエリを w (実運用での質問回数、仮説は 1) で重み付けした割合。実需が満たされているかを見る主指標。
plan 成立率 / 修復リトライ
plan 成立率: クエリをスキーマで構造的にプラン化できた率 (失敗時は全文検索に退避)。 常時 100% が正常で、低下はスキーマ/プロンプト退行のアラート (番犬指標)。 修復リトライ: LLM 出力が検証に失敗したとき、エラー内容を添えて再生成させる 仕組み (タイムラインの黄枠)。

intent × field 結合マトリクス

朱枠のセル = スキーマ設計時に「このフィールドはこの intent を支える」と宣言された組 (= serves_intents 宣言)。上段の数値 = その intent のクエリの実行プランでフィールドが 実際に使われた回数、下段の小さい数値 = 全クエリ横断での使用回数。 朱枠なのに 0 のセルは「宣言だけで実際には使われていない」結合不全の兆候。
