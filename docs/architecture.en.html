Agent-Native RLM Architecture

A bird's-eye view of the self-improving loop formed by cu-agent-rlm and qu-agent-rlm. The diagrams show how the catalog / prompts / queries pillars evolve as first-class state, while a strict tool boundary keeps raw transcripts out of every LLM prompt.

03 — Tool Boundary

7 step tools + 5 base tools

The LLM can only touch a fixed set of tools. call_records itself never enters a prompt; every reference is mediated by chunk:<id>.

Retrieval — agent-callable

QU planner picks the tool per intent

  • bm25_search_chunks — exact terms, names, IDs (bm25s + numpy)
  • embedding_search_chunks — paraphrase, semantic recall
  • search_chunks — abstract / mode-routed (lexical / bm25 / embedding / hybrid)
  • fetch_chunks — evidence resolver (always called before a claim)
Silver — structured access

Operate on values without touching raw transcripts

  • query_silver — filter by promoted fields
  • aggregate_silver — group_by on aggregatable fields
  • run_sql — Databricks SELECT (contract only)

04 — Promotion Gate Pattern

Proposal → gate → state

Catalog / prompts / queries all follow the same proposal → gate → accepted state pattern. Having a gate is what draws the line between "code" and "data".

08 — Design Principles

10 things this architecture believes

  1. Tool boundary is fixed at five tools. Raw transcripts never reach an LLM; all references go through chunk:<id>.
  2. All three pillars share versioned identity + promotion gate. Catalog / prompts / queries follow the same lifecycle.
  3. Every LLM path carries validator + fallback + provenance. Trace shows <provider>->fallback:<base>; prompt_id + hash; composite provenance strings.
  4. frozen=True is a type-level guarantee against auto-mutation. Changes require a new instance (a version bump).
  5. Bake PROMOTION_GATE into the data. "Why we don't promote" is recorded in the artifact itself.
  6. Operationally separate self-eval bias. Judge ↔ planner model separation; synthetic queries stay outside promotion gates.
  7. LLM-first + reliability fallback. Semantic decisions belong to the LLM; heuristics only catch validation/API failures and support smoke tests.
  8. Append-only signal logs. column_requests / prompt_repair_request / orchestration_trace accumulate over time and are aggregated downstream.
  9. Keep deterministic control. BM25 explainability, query diversity, forced search iteration, and smoke eval are retained as reliability infrastructure.
  10. Observability via structured JSON. Trace tool names expand backend / provider / fallback chain into strings.
